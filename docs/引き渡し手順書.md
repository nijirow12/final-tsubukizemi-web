## 引き渡しパターンの選択

本プロジェクトでは、パターンBでいきます。

### パターンA: 完全移譲型（クライアント単独運用）

- 全サービス・リポジトリの所有権をクライアントに移譲
- 開発者は引き渡し後、コードに関与しない
- クライアント側にエンジニアがいる場合、または今後の改修予定がない場合に適する

### パターンB: コード保守継続型（推奨・本プロジェクトのデフォルト）

- **コンテンツ管理はクライアント、コード管理は開発者が継続**
- GitHub リポジトリはクライアント所有だが、開発者を Collaborator (Admin) として維持
- Vercel はクライアント所有、`master` への push で自動デプロイされるため、開発者はローカルから `git push` するだけで改修を反映できる
- クライアントは microCMS / Google Drive / Instagram でコンテンツを更新し、コードには一切触れない
- **コスト: 月額 ¥0**（GitHub Free + Vercel Hobby）

> **Vercel Hobby プランの注意:** Hobby プランは規約上「非商用利用のみ」です。大学ゼミの教育目的サイトなら問題ありませんが、広告収益や有料サービス化する場合は Pro プラン（$20/月〜）への移行が必要です。

**本書は以降、パターンB（推奨）を前提とした手順を記載し、パターンA特有の作業は `【パターンA限定】` タグで注記します。**

## 2. GitHub（ソースコード管理）

### 2.1 現状

- **リポジトリ:** `https://github.com/nijirow12/final-tsubukizemi-web`
- **ブランチ:** `master`（本番ブランチ）
- **デフォルトでの動作:** `master` に push すると Vercel が自動デプロイします

### 2.2 引き渡し手順（パターンB: コード保守継続型）

**想定する最終状態:**
- リポジトリのオーナー = クライアント（ゼミ共用 GitHub アカウント）
- Collaborator (Admin) = 開発者 — `master` へ push する権限を保持

#### 手順

1. クライアントは GitHub アカウントを作成（既存なら可。ゼミ共用アカウント推奨）
2. 開発者がリポジトリの **Settings → General → Transfer ownership** から、クライアントのアカウントへ移譲
3. クライアントは Transfer を承認
4. 移譲後、クライアントが **Settings → Collaborators** から開発者を招待し、`Admin` 権限を付与
5. 開発者は招待メールを承認
6. ローカルの git remote を更新

   ```bash
   git remote set-url origin https://github.com/【クライアントのアカウント名】/final-tsubukizemi-web.git
   git pull origin master
   ```

7. 動作確認: 開発者がローカルで軽微な変更を加えて `git push origin master` し、Vercel が自動デプロイされることを確認


### 2.3 必要な権限

**開発者側（Collaborator）:**
- コードの閲覧・編集・push
- ブランチ作成・PR 作成

**クライアント側（Owner）:**
- Settings 変更（Collaborator の追加/削除、リポジトリ削除）
- Secrets 管理（GitHub Actions を使う場合）
- Transfer ownership
- Billing（Free プラン継続の確認）

## 3. Vercel（ホスティング）

### 3.1 現状

- **プロジェクト名:** `final-tsubukizemi-web`
- **プラン:** Hobby（無料）
- **デプロイ設定:** `master` ブランチに push すると自動デプロイ
- **Node.js:** 24.x
- **本番URL:** Vercel ダッシュボードの `Domains` タブで確認

### 3.2 引き渡し手順（パターンB: コード保守継続型）

**想定する最終状態:**
- Vercel プロジェクトのオーナー = クライアント
- GitHub 連携 = クライアント所有のリポジトリ
- **開発者がローカルから `git push origin master` すれば、クライアントの Vercel が自動でビルド・デプロイする**

> この構成が成立するのは、Vercel のデプロイトリガーが **リポジトリへの push イベント** だからです。誰が push したかに関係なく、リポジトリが更新されれば Vercel が反応します。つまり開発者は Vercel のアカウント権限を持たなくても、GitHub Collaborator 権限だけで本番反映が可能です。

#### A. クライアントの Vercel アカウント作成

1. `https://vercel.com` にアクセス
2. **「Continue with GitHub」** で、クライアントの GitHub アカウントでサインアップ
3. 無料プラン（**Hobby**）を選択

#### B. プロジェクトの移譲

**方法1: プロジェクト Transfer（推奨）**

1. 開発者が `Vercel Dashboard → final-tsubukizemi-web → Settings → General`
2. 一番下の **「Transfer Project」** をクリック
3. 移譲先（クライアントのアカウント）を指定
4. クライアント側で承認

**方法2: 一度削除して再インポート**（Transfer が使えない場合）

1. 開発者が Vercel からプロジェクトを削除（コードは GitHub に残るので安全）
2. クライアントが自分の Vercel アカウントから `Import Project` → GitHub リポジトリ（§2で移譲済み）を選択
3. Framework: **Next.js**、Node: **24.x** を指定してインポート
4. 環境変数を設定（§3.3）

#### C. 動作確認

- クライアントの Vercel ダッシュボードに `final-tsubukizemi-web` が表示されていること
- 開発者がローカルで試しに1行変更して `git push origin master` し、クライアントの Vercel ダッシュボードに新しいデプロイが自動生成されることを確認

> **【パターンA限定】** 開発者も関与しない場合は C の動作確認は不要ですが、代わりに運用担当者自身で「GitHub 編集 → Vercel 自動デプロイ」の流れを一度試しておくことを推奨します。

### 3.3 環境変数の設定（**最重要**）

Vercel Dashboard → プロジェクト → **Settings → Environment Variables** から以下を **すべて** 登録してください。対象環境は `Production` / `Preview` / `Development` の3つ全てにチェックを入れます。

| 変数名 | 値（開発者から別途受領） | 種別 |
|--------|----------------------|------|
| `MICROCMS_SERVICE_DOMAIN` | `tsubukizemi`（または新しいサービス名） | Secret |
| `MICROCMS_API_KEY` | microCMS で発行したAPIキー | Secret |
| `GOOGLE_DRIVE_ROOT_FOLDER_ID` | Google Drive ルートフォルダのID | Secret |
| `GOOGLE_SERVICE_ACCOUNT_JSON` | サービスアカウントJSONファイルの中身（文字列化） | Secret |
| `NEXT_PUBLIC_BEHOLD_FEED_ID` | Behold.so で発行した Feed ID | Public |

> **重要:** `NEXT_PUBLIC_` で始まる変数のみブラウザ側に公開されます。他の変数は絶対に `NEXT_PUBLIC_` プレフィックスを付けないでください。

設定後、**「Redeploy」** で再デプロイして有効化します。

### 3.4 独自ドメイン設定（任意）

1. お名前.com / Google Domains 等でドメイン取得
2. Vercel Dashboard → プロジェクト → **Settings → Domains**
3. ドメインを追加し、表示される DNS レコード（A レコードまたは CNAME）を取得
4. ドメインレジストラ側で DNS レコードを設定
5. 数分〜24時間で SSL 証明書が自動発行されます

## 4. Google Cloud Platform（Drive API）

### 4.1 なぜ必要か

本サイトの「活動」ページでは、Google Drive に格納された国別の活動写真を自動取得して表示しています。そのために **Google Cloud のサービスアカウント** が必要です。

### 4.2 新しいサービスアカウント作成手順

開発者の既存サービスアカウントは廃止し、運用担当者の Google アカウントで新規発行することを推奨します。

1. `https://console.cloud.google.com/` にアクセス
2. 画面上部の **プロジェクト選択 → 新規プロジェクト**
   - プロジェクト名例: `tsubukizemi-web`
3. **APIとサービス → ライブラリ** で **「Google Drive API」** を検索 → **有効化**
4. **APIとサービス → 認証情報 → 認証情報を作成 → サービスアカウント**
   - サービスアカウント名: `tsubukizemi-drive-reader`
   - ロール: 付与しない（Driveへの共有で権限付与するため）
5. 作成後、該当サービスアカウントをクリック → **キー → 鍵を追加 → 新しい鍵を作成 → JSON**
6. JSON ファイルがダウンロードされます。**このファイルは再発行不可なので厳重保管してください。**

### 4.3 Vercel への登録

ダウンロードした JSON ファイルの**中身全体**（`{` から `}` まで）をコピーし、Vercel の環境変数 `GOOGLE_SERVICE_ACCOUNT_JSON` に貼り付けます。

> **注意:** 改行を含む長い文字列ですが、Vercel はそのまま貼り付け可能です。クオートで囲む必要はありません。

### 4.4 廃止するサービスアカウント

開発時に使われていた古いサービスアカウント（`tsubukizemi-demo-*`）は、引き渡し完了後に Google Cloud Console から **削除** してください。

## 5. Google Drive（写真格納）

### 5.1 フォルダ構造（厳守）

サイト側は `ルート → 地域 → 国` の **2階層** を前提にフォルダを自動探索します。この構造を崩すと写真が取得できなくなります。

```
ルートフォルダ (例: 津吹ゼミ活動写真)    ← GOOGLE_DRIVE_ROOT_FOLDER_ID で指定
├── Asia/                              ← 第1階層: 地域（任意の名前でOK）
│   ├── 🇰🇭カンボジア/                  ← 第2階層: 国名を含むフォルダ
│   ├── 🇻🇳ベトナム/
│   ├── 🇹🇭タイ/
│   └── ... （国ごと）
└── ... （地域ごと）
```

#### 国名の命名ルール

国フォルダ名には、以下の **日本語または英語の国名** を含める必要があります（絵文字や記号は無視されます。大文字小文字も問いません）。

| 日本語国名 | 英語国名 |
|----------|---------|
| 日本 | japan |
| カンボジア | cambodia |
| ベトナム | vietnam |
| タイ | thailand |
| フィリピン | philippines |
| インドネシア | indonesia |
| マレーシア | malaysia |
| シンガポール | singapore |
| ミャンマー | myanmar |
| ラオス | laos |
| インド | india |

> 例: `🇰🇭カンボジア`, `cambodia`, `2025-Cambodia-Field-Work` などはすべて `cambodia` と判定されます。
>
> 逆に `プノンペン` や `アンコールワット` といったフォルダ名では判定できません。必ず国名を含めてください。

### 5.2 引き渡し手順

1. 開発者の Google Drive にある既存フォルダを、運用担当者の Google アカウントへ**所有権移譲**
   - 既存フォルダを右クリック → **共有** → 運用担当者のアカウントを追加
   - **「オーナー権限を譲渡」** を選択
2. 運用担当者は受領後、**ルートフォルダのID** を控える
   - Drive でフォルダを開いた時の URL: `https://drive.google.com/drive/folders/【ここがID】`
3. このIDを Vercel の環境変数 `GOOGLE_DRIVE_ROOT_FOLDER_ID` に登録
4. **重要:** §4 で作成したサービスアカウントのメールアドレス（`xxxx@yyyy.iam.gserviceaccount.com`）を、ルートフォルダに **「閲覧者」権限** で共有する
   - この共有を忘れると Drive API が 404 エラーを返します

## 6. microCMS（コンテンツ管理）

### 6.1 用途

メンバー情報など、コード修正なしで更新したいコンテンツを管理します。

- **Vercel の環境変数 `MICROCMS_SERVICE_DOMAIN` と `MICROCMS_API_KEY` が設定済みの場合**: microCMS から自動取得されます
- **未設定の場合**: デモデータ（[src/data/members-demo.ts](../src/data/members-demo.ts)）が表示されます

**コード変更は不要です。** 環境変数を設定して Redeploy するだけで切り替わります。

> **注意:** 初期設計にあった `news` API と `activity-photos` API は実装していません。活動写真は Google Drive + Instagram（Behold.so）の二系統で管理しています。

### 6.2 アカウント開設

1. `https://microcms.io/` にアクセス
2. 運用担当者のメールアドレスで登録
3. 新規サービスを作成
   - サービス名例: `tsubukizemi`
   - サービスドメイン: `tsubukizemi`（以降、URL に使用）
4. プラン: **Hobby（無料）** を選択

### 6.3 API 作成

**API名:** `tsubukizemi`（メンバー情報）
**API種別:** リスト形式
**エンドポイントID:** `tsubukizemi`

#### スキーマ（実際の microCMS フィールドID）

| フィールドID          | 表示名       | 種別                       | 必須  |
| ---------------- | --------- | ------------------------ | --- |
| `nameJP`         | 氏名（日本語）   | テキスト                     | ✅   |
| `nameEN`         | 氏名（英語）    | テキスト                     | ✅   |
| `introductionJP` | 自己紹介（日本語） | テキストエリア                  | ✅   |
| `introductionEN` | 自己紹介（英語）  | テキストエリア                  | ✅   |
| `projectsJP`     | 活動内容（日本語） | テキストエリア                  |     |
| `projectsEN`     | 活動内容（英語）  | テキストエリア                  |     |
| `countries`      | 訪問国       | 複数値                      |     |
| `socialurl`      | SNS URL   | テキスト                     |     |
| `photo`          | 写真        | 画像                       |     |
| `grade`          | 学年        | セレクト（1年/2年/3年/4年/alumni） | ✅   |

> **`grade` の `alumni` について:** `alumni` を選択した場合、サイト上では OB/OG として表示され、学年は非表示になります。

### 6.4 API キー発行

1. microCMS Dashboard → **サービス設定 → API キー**
2. デフォルトキーは削除し、**新規キーを発行**
3. 権限: `GET` のみチェック
4. 発行されたキーを Vercel の `MICROCMS_API_KEY` に登録
5. `MICROCMS_SERVICE_DOMAIN` にはサービス名（例: `tsubukizemi`）を登録

## 8. ローカル開発環境の構築（担当者向け）

将来、運用担当者がコードを編集する可能性がある場合は、以下の手順でローカル環境を構築してください。

### 8.1 必須ツール

- **Node.js 24.x** （`https://nodejs.org/`）
- **Git** （`https://git-scm.com/`）
- **VS Code** などのエディタ（推奨）

### 8.2 セットアップ

```bash
# 1. リポジトリをクローン
git clone https://github.com/nijirow12/final-tsubukizemi-web.git
cd final-tsubukizemi-web

# 2. 依存パッケージをインストール
npm install

# 3. 環境変数ファイルを作成
touch .env.local
```

`.env.local` に以下を記載:

```env
MICROCMS_SERVICE_DOMAIN=tsubukizemi
MICROCMS_API_KEY=（microCMSのAPIキー）
GOOGLE_DRIVE_ROOT_FOLDER_ID=（Driveフォルダ ID）
GOOGLE_SERVICE_ACCOUNT_JSON={"type":"service_account",...}
NEXT_PUBLIC_BEHOLD_FEED_ID=（Behold Feed ID）
```

```bash
# 4. 開発サーバー起動
npm run dev
```

ブラウザで `http://localhost:3000` を開いて確認。

### 8.3 本番デプロイ

```bash
git add .
git commit -m "変更内容の説明"
```
