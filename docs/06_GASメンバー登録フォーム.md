## 1. 背景と目的

**背景:**
津吹ゼミ公式サイトのメンバーページ（`/members`）は、現状デモデータか microCMS のデータを描画している。運用でこれを更新する際、管理者が microCMS 管理画面で手動入力すると、日英併記・所属情報・プロジェクト紹介などを毎回手打ちする負担が大きい。

**目的:**

- **入力の分散化:** ゼミ生本人が Google フォームで入力し、管理者は確認・公開判断に集中できるようにする
- **二言語担保:** 公開ページで日本語・英語切替を破綻なく出せるよう、フォームの時点で両方を強制的に集める
- **写真の自動保全:** プロフィール写真は Google Drive に自動保存し、紛失を防ぐ
- **拡張前提:** 将来 microCMS / Slack 通知 / 別 DB への連携を追加できるよう、回答データを構造化しておく

## 2. 関係者

| 役割 | 説明 |
|---|---|
| **回答者（ゼミ生）** | フォームに自分の情報を入力する現役ゼミ生 / OB・OG |
| **管理者（ゼミ運営）** | スプレッドシートと microCMS を確認し、公開可否を判断してサイト反映を行う担当 |
| **サイト訪問者** | 公開後の結果を `/members` で閲覧するエンドユーザー |

## 3. フォーム仕様

### 3.1 フォーム名

> **津吹ゼミ メンバー登録フォーム**

### 3.2 フォーム説明（冒頭に表示する文）

> 津吹ゼミ公式サイトに掲載するメンバー情報を登録するフォームです。日本語と英語の両方を入力してください（英語は簡単な翻訳でも構いません）。送信内容はサイトのメンバーページに掲載されます。
>
> - **所要時間の目安:** 約 10 分
> - **写真:** 正方形に近いサイズ（1:1 推奨）、1MB 以下推奨
> - **SNS:** Instagram / LinkedIn / X など任意の 1 つをリンク可（省略可）
>
> 修正が必要な場合はサイト管理者までご連絡ください。

### 3.3 質問項目

| # | 質問タイトル | 回答形式 | 必須 | データキー |
|---|---|---|:---:|---|
| 1 | 名前（日本語） | 記述式（短文） | ✅ | `nameJP` |
| 2 | Name (English) | 記述式（短文） | ✅ | `nameEN` |
| 3 | 自己紹介（日本語） | 記述式（長文） | ✅ | `introductionJP` |
| 4 | Self-introduction (English) | 記述式（長文） | ✅ | `introductionEN` |
| 5 | 携わったプロジェクト（日本語） | 記述式（長文） |  | `projectsJP` |
| 6 | Projects (English) | 記述式（長文） |  | `projectsEN` |
| 7 | 活動国 | 記述式（短文 / 自由記述） | ✅ | `countries` |
| 8 | 学年 | ラジオボタン（単一選択） | ✅ | `grade` |
| 9 | SNS URL | 記述式（短文） |  | `socialurl` |
| 10 | プロフィール写真 | ファイルのアップロード（1 枚 / 画像のみ） |  | `photo` |

### 3.4 選択肢・入力フォーマット

**活動国（Q7 / 自由記述）:**

- 複数国ある場合は区切り文字で列挙: カンマ `,` / 読点 `、` / スラッシュ `/` / 改行 のいずれか
- 日本語 / 英語どちらで書いてもよい（例: `カンボジア, ベトナム` / `Cambodia / Vietnam` / `タイ、Thailand`）
- 後段処理で **配列化して保存する**（[§4.1 参照](#41-回答オブジェクト形状)）
- サイト地球儀ピンと一致させる場合は、[src/data/pin-coordinates.ts](../src/data/pin-coordinates.ts) の `labelJa` / `label` に揃った名称で入力することを推奨

**学年（Q8 / 単一選択）:**

```
1年 / 2年 / 3年 / 4年 / 5年 / 6年 / 7年 / alumni（OB・OG）
```

- `alumni` を選んだ場合、サイト側で自動的に OB/OG 扱い（`isOB = true`）となり学年表示は消える
  （判定ロジック: [src/lib/microcms.ts:67](../src/lib/microcms.ts#L67)）

**プロフィール写真（Q10 / ファイルアップロード）:**

- 画像ファイル（PNG / JPG）のみ
- 1 ファイル上限
- 回答にはファイルの **Google Drive ファイル ID** が入る（回答者が Google ログイン必須）
- 後段処理で Drive URL に変換して保管する想定

## 4. 回答データ仕様

### 4.1 回答オブジェクト形状

フォーム送信を受ける後段スクリプトが、最終的に扱うべき構造化オブジェクトは以下を想定する。フォームの生回答（文字列 / 配列）を、スクリプトで以下の形に整形することを要件とする。

```json
{
  "timestamp":      "2026-04-22T05:23:45.000Z",
  "nameJP":         "山田 太郎",
  "nameEN":         "Taro Yamada",
  "introductionJP": "経済学部4年の山田です...",
  "introductionEN": "I'm Taro, a 4th year student...",
  "projectsJP":     "カンボジアの教育支援プロジェクト...",
  "projectsEN":     "Educational support project in Cambodia...",
  "countries":      ["カンボジア", "ベトナム"],
  "grade":          "3年",
  "socialurl":      "https://instagram.com/taro_yamada",
  "photo":          ["https://drive.google.com/file/d/XXXXXXXXX/view"]
}
```

### 4.2 正規化ルール

| フィールド | 元データ | 正規化ルール |
|---|---|---|
| `timestamp` | フォームイベント時刻 | ISO 8601 文字列（UTC） |
| `nameJP` / `nameEN` | 文字列 | trim のみ |
| `introductionJP` / `introductionEN` | 複数行文字列 | 改行はそのまま保持 |
| `projectsJP` / `projectsEN` | 複数行文字列 | 改行はそのまま保持 |
| `countries` | 自由記述文字列 | 区切り文字（`,` / `、` / `/` / 改行）で分割 → trim → 空要素除去 → **配列**化 |
| `grade` | 選択肢 | 文字列のまま（例: `"3年"`, `"alumni"`） |
| `socialurl` | 文字列 | trim のみ。未入力なら `null` |
| `photo` | Drive file ID 配列 | 各 ID を `DriveApp.getFileById(id).getUrl()` で Drive URL に変換した **配列** |

### 4.3 欠損値の扱い

- 必須項目（Q1–Q4 / Q7 / Q8）は Google フォーム側で「必須」にして空送信を禁止する
- 任意項目（Q5 / Q6 / Q9 / Q10）の未入力は、出力オブジェクトから **キーごと省略** するか `""` / `null` で埋めるか、後段実装の都合で一貫させる

## 5. 保管先

**回答の一次保管先はスプレッドシート**（Google フォームの純正「回答」→「スプレッドシート連携」で作成）。

- フォームの「回答」タブ → スプレッドシートアイコンから **新規作成** または既存選択
- Google フォームが全回答を自動で 1 行ずつ追記する（追加スクリプト不要）
- 管理者はこのスプレッドシートを「入力された生データの確認窓口」として使う

後段スクリプトが整形オブジェクト（§4）を別シート / 別スプレッドシート / 外部 API へ送ることは **別要件**。本書のスコープ外。

## 6. 通知要件

| 事象 | 通知方法 | 設定箇所 |
|---|---|---|
| 新規回答 | 管理者にメール | Google フォーム「回答」タブ → 「⋮」→ **新しい回答のメール通知を受け取る** を ON |
| 後段スクリプトの実行失敗 | スクリプト所有者にメール（Apps Script 既定） | 追加設定なし。スクリプトが throw すれば自動送信 |

**上記以外のカスタム通知（Slack 等）は当面スコープ外。** 必要になった時点で後段スクリプトに実装する。

---

## 7. サイト反映

- 現時点では **自動反映しない**（手動運用）
- 管理者がスプレッドシートで内容を確認 → 問題なければ microCMS 管理画面で当該メンバーのエントリを作成 / 更新 → 公開
- サイト側は ISR（`revalidate = 300`）で 5 分以内に反映（[src/app/api/members/route.ts](../src/app/api/members/route.ts)）
- 将来「スプレッドシート → microCMS 自動連携」を追加したくなったら、後段スクリプトで `collectFormItems` の出力を microCMS Content API に POST する実装を足す（拡張要件）

## 8. 非機能要件

| 項目 | 要件 |
|---|---|
| **機密性** | フォーム回答・写真は Google Workspace の既定範囲（フォーム所有アカウント配下）にのみ保存する。外部共有しない。 |
| **可用性** | Google フォーム / Google スプレッドシートのサービス稼働率に従う（個別 SLA は設けない） |
| **保守性** | 質問タイトルは「データキー」と明確に対応させ、フォーム改修時は本書 §3.3 とデータキー仕様を同時更新する |
| **拡張性** | 後段スクリプトは「回答 → 構造化オブジェクト化」を 1 つの関数（`collectFormItems` 相当）に閉じ込め、そこから先の連携は差し替え可能にする |
| **プライバシー** | フォーム冒頭の説明文で「サイトに掲載される」旨を明示する（§3.2 準拠） |
| **二言語対応** | 日本語 / 英語の両方を必須で集める。英語の自動翻訳は行わない（本人記入が原則） |

---

## 9. 変更時の注意

- **質問タイトルの変更:** 後段スクリプトの `FIELD_MAP`（質問タイトル → データキー）も同時に更新。タイトルがズレると該当項目が出力オブジェクトに載らなくなる（エラーではなく「黙って欠損」するため検知しづらい）
- **新しい活動国を追加したとき:** サイトの地球儀ピン表示には別途 [src/data/pin-coordinates.ts](../src/data/pin-coordinates.ts) への座標追加が必要
- **データキーの変更:** microCMS 側スキーマ（`nameJP` / `nameEN` 等）と整合を取る必要があるため、単独変更はしない（[src/lib/microcms.ts](../src/lib/microcms.ts) の `RawMember` 型を参照）
- **必須/任意の切り替え:** §3.3 と §4.3 を同時更新すること
